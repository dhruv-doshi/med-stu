# Implementation Plan: Vaani BYOL Live Virtual Patient MVP

**Feature**: `001-local-vaani-mvp` | **Date**: 2026-07-18 | **Spec**: [spec.md](spec.md)

## Summary

Deliver phone and in-browser WebRTC clinical simulations with Vaani as the native real-time voice runtime and FastAPI as the Bring Your Own LLM (BYOL) server. Vaani owns STT, TTS, call media, and model-native turn-taking; FastAPI owns case state, permitted facts, investigations, and evaluation. In the preferred no-phone flow, the browser is both the LiveKit media participant and the live dashboard.

## Architecture

```text
Browser dashboard ── WebSocket ── FastAPI consultation service
                                     │
                                     ├─ case/state engine (authoritative)
                                     ├─ OpenRouter Patient Agent
                                     ├─ action extractor + investigation service
                                     └─ Evaluator Agent
                                     │  persistent BYOL WebSocket per call
Vaani phone agent ───────────────────┘
  STT / TTS / call media / native interruption
                                     │
                                Vaani lifecycle webhooks

Browser-live fallback ── browser STT/TTS ── same FastAPI turn/action API

Browser Vaani call ── LiveKit WebRTC token ── Vaani STT/TTS/turn-taking
```

## Technical decisions

- **Vaani setup**: Configure one Vaani agent with BYOL enabled and a no-platform-LLM fallback. The agent calls `wss://<public-host>/vaani/byol/<call_id>`; Vaani appends the call ID.
- **Call mapping**: Browser creates a consultation; FastAPI dispatches the Vaani call with the selected case metadata, records the returned call ID, and maps it to the consultation before serving BYOL turns.
- **Patient Agent**: For every `response_required` message, use the most recent learner turn plus stored consultation state. OpenRouter first selects permitted fact IDs, then writes a short patient response. Stream approved text chunks using Vaani’s `response_id` protocol.
- **Clinical actions**: Before the patient response, extract structured actions from the learner turn. The State Engine validates orders/differentials against the selected case, mutates state, and broadcasts browser events. Investigation values are never generated by an LLM.
- **Live dashboard**: A browser WebSocket receives `call_status`, `transcript_turn`, `investigation_ordered`, `investigation_result`, `consultation_completed`, and `evaluation_ready` events.
- **Call end**: Vaani webhook identifies `call_ended` then `call_postprocessing`; FastAPI stores final transcript, runs deterministic scoring plus Evaluator Agent feedback, and emits `evaluation_ready`.
- **Barge-in**: Vaani is the audio/turn-taking authority. The backend never starts a second response after a newer Vaani `response_required` turn; it cancels any outstanding generation task for that call before handling the new response ID.
- **Free browser demo**: Chrome-compatible Web Speech APIs provide continuous local microphone capture and browser TTS. A recognised learner utterance cancels active browser TTS and uses `input_mode: browser_live`, which reuses the safe action/extraction path. This is explicitly a local fallback rather than model-native audio handling.
- **Vaani WebRTC**: FastAPI calls `/api/trigger-call/` with `medium: "webrtc"` and `X-Agent-Id` set to the consultation ID. It maps the returned `room_name`, returns the short-lived token/connection URL, and the browser joins with the LiveKit client. The long-lived API key never reaches the browser.

## Security and failure policy

- Store `VAANI_API_KEY` and `OPENROUTER_API_KEY` in server `.env` only. Do not expose either in Next.js.
- Verify Vaani webhook signatures when the dashboard supplies its signing secret; reject stale/replayed requests.
- Configure Vaani BYOL fallback as **No fallback** so a platform model cannot invent clinical facts.
- On BYOL disconnect, broadcast a degraded status; retain the browser text workflow. On call end, evaluate only the transcript that Vaani provides.

## Delivery sequence

1. Add Vaani configuration, dispatch endpoint, call-to-consultation map, and dashboard event broker.
2. Implement Vaani BYOL handshake, `response_required` handling, streamed replies, cancellation, and action extraction.
3. Add webhook receipt, final-transcript reconciliation, evaluation trigger, and idempotency tests.
4. Update the browser to start a call, subscribe to live events, and render call/transcript/order/evaluation status.
5. Configure Vaani dashboard, expose local server with WSS, run the chest-pain script, and validate barge-in plus live ECG result.

## Deferred production work

Persist sessions/call maps in PostgreSQL and Redis; deploy WSS-capable services; add authentication, audit retention, signed webhook verification, observability, cost controls, and multi-region recovery.
